# 外部模型接口设计

本文档记录 `frac_dete_demo` 为后续接入其它部位骨折模型、骨结构分割模型或外部参考模型预留的接口设计。

## 1. 设计目标

当前 demo 暂时没有可直接用于腕、足、踝、肘等部位的可靠骨折检测模型。

后续可能接入的模型包括：

- 肋骨骨折 CT 模型；
- 颈椎/脊柱骨折模型；
- 骨结构分割模型；
- 医院数据训练出的本项目模型；
- 其它候选生成器。

这些模型的输出形式可能不同，有的是分类，有的是分割，有的是检测框，有的是点位候选。为了避免后续在 `app.py` 中写大量临时逻辑，当前先预留统一 adapter 接口。

## 3. 统一输出格式

所有外部模型或候选生成器最终都应转换为 `ExternalCandidate`：

```python
ExternalCandidate(
    source="heuristic",
    candidate_id="1_edge_0492",
    case_id="1",
    slice_index=492,
    x=147,
    y=52,
    z=492,
    score=0.82,
    label="weak_candidate",
    note="sharp bone-window edge change",
    status="unreviewed",
)
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `source` | 候选来源，例如 `heuristic`、`ribfrac_reference` |
| `candidate_id` | 候选点唯一编号 |
| `case_id` | 病例编号 |
| `slice_index` | 轴位切片号 |
| `x` | 图像左右方向坐标 |
| `y` | 图像上下方向坐标 |
| `z` | 体数据 z 坐标 |
| `score` | 当前来源给出的排序分或参考分 |
| `label` | 候选标签或模型输出类别 |
| `note` | 对候选来源、限制或原因的说明 |
| `status` | 人工审核状态 |

注意：不同模型的 `score` 不一定可直接比较。跨模型融合前需要校准。

## 4. Adapter 协议

每个 adapter 应实现：

```python
class SomeAdapter:
    name = "some_source"
    display_name = "Some Source"
    enabled = True
    out_of_domain = False

    def predict(self, case_id, volume_hu, metadata=None):
        ...
        return list[ExternalCandidate]
```

其中：

- `name`：程序内部使用的来源名；
- `display_name`：页面或日志中展示的名称；
- `enabled`：是否默认启用；
- `out_of_domain`：是否属于跨部位参考模型；
- `predict()`：接收 CT 体数据，返回统一候选点列表。

## 5. 当前已实现 Adapter

### 5.1 HeuristicAdapter

文件：

```text
src/external_models/heuristic_adapter.py
```

状态：


作用：

- 将 `Candidate` 转换为 `ExternalCandidate`；
- 当前页面实际展示的候选点仍来自它。

## 6. Registry

文件：

```text
src/external_models/registry.py
```

主要函数：

```python
get_all_adapters()
get_enabled_adapters()
run_enabled_adapters(case_id, volume_hu, metadata=None)
```

当前 `app.py` 不再直接调用 `detect_candidates()`，而是调用：

```python
run_enabled_adapters(...)
```

这样后续新增模型时，只需要：

1. 新建 adapter 文件；
2. 在 `registry.py` 中注册；
3. 设置 `enabled = True` 或在页面中提供开关。

## 7. 后续接入外部模型的建议

如果接入肋骨或脊柱模型，不建议把输出当作可靠诊断结果，而应标注为：

```text
out_of_domain_reference
```

推荐输出说明：

```python
note="Rib fracture model reference; not trained for wrist/foot CT."
```

页面上应单独提示：

> 该结果来自跨部位参考模型，不代表当前部位可靠诊断。

## 8. 后续可能的页面改造

当前页面仍使用原来的 `Weak Candidates` 表格。

后续可以扩展为：

- 增加 `source` 列；
- 增加 `label` 列；
- 增加模型来源筛选；
- 增加启用/关闭 adapter 的复选框；
- 将跨域参考模型输出放到 `External Model References` 区域；
- 区分 `heuristic`、`manual`、`external_reference`、`trained_model`。

## 9. 当前版本的边界

当前只是预留接口和保存架构版本：

- 没有下载或集成任何外部模型权重；
- 没有运行肋骨/脊柱模型；
- 页面输出仍主要来自启发式规则；
- 外部模型占位 adapter 默认关闭；
- 还没有做跨模型分数校准。

这一步的价值是：后续可以按同一 adapter 规范逐步接入外部参考模型，而不破坏现有 demo 主流程。

### 10.3 关键阻塞

目前有两个实际阻塞。

第一，最适合手腕、足部、踝、肘等部位的任务是：

```text
appendicular_bones
```

该任务包含更相关的四肢骨结构，例如：

- `tibia`
- `fibula`
- `tarsal`
- `metatarsal`
- `ulna`
- `radius`
- `carpal`
- `metacarpal`

但 TotalSegmentator 当前提示该任务不是开放任务，需要 license。没有 license 前，不能直接作为免费外部专家接入主 demo。

第二，开放的 `total` 任务可以下载权重，但推理阶段在当前 Windows 环境中仍以原生退出码退出：

```text
-1066598273
```

没有 Python traceback。该问题发生在实际推理阶段，不是 DICOM 读取、NIfTI 导出或 Python import 阶段。

### 10.4 当前结论

TotalSegmentator 暂时不接入主页面。

原因：

- `appendicular_bones` 需要 license；
- 开放 `total` 任务对腕骨、跗骨、掌骨、跖骨等小骨覆盖不足；
- 当前 Windows 推理链路仍有 native crash；
- 强行接入会把不稳定依赖带进 demo 主流程。

### 10.5 后续可恢复条件

满足以下任一条件后，可以恢复 TotalSegmentator adapter 开发：

- 申请并配置 TotalSegmentator academic license，重新测试 `appendicular_bones`；
- 在 WSL2 / Linux / Docker 中跑通 TotalSegmentator；
- 找到 Windows native crash 的稳定修复方案；
- 只使用 TotalSegmentator 的开放大骨任务，并明确标注其不覆盖手足小骨。

如果恢复开发，建议 adapter 命名为：

```text
BoneSegmentationReferenceAdapter
```

并且默认只作为骨结构参考，不输出骨折诊断结论。
