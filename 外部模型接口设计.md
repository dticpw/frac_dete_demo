# 外部模型接口设计

本文档记录 `frac_dete_demo` 为后续接入其它部位骨折模型、骨结构分割模型或外部参考模型预留的接口设计。

## 1. 设计目标

当前 demo 暂时没有可直接用于腕、足、踝、肘等部位的可靠骨折检测模型。

后续可能接入的模型包括：

- 肋骨骨折 CT 模型；
- 颈椎/脊柱骨折模型；
- 骨结构分割模型；
- 医院数据训练出的本项目模型；
- 其它候选生成器。

这些模型的输出形式可能不同，有的是分类，有的是分割，有的是检测框，有的是点位候选。为了避免后续在 `app.py` 中写大量临时逻辑，当前先预留统一 adapter 接口。

## 2. 当前目录结构

相关代码位于：

## 3. 统一输出格式

所有外部模型或候选生成器最终都应转换为 `ExternalCandidate`：

```python
ExternalCandidate(
    source="heuristic",
    candidate_id="1_edge_0492",
    case_id="1",
    slice_index=492,
    x=147,
    y=52,
    z=492,
    score=0.82,
    label="weak_candidate",
    note="sharp bone-window edge change",
    status="unreviewed",
)
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `source` | 候选来源，例如 `heuristic`、`ribfrac_reference` |
| `candidate_id` | 候选点唯一编号 |
| `case_id` | 病例编号 |
| `slice_index` | 轴位切片号 |
| `x` | 图像左右方向坐标 |
| `y` | 图像上下方向坐标 |
| `z` | 体数据 z 坐标 |
| `score` | 当前来源给出的排序分或参考分 |
| `label` | 候选标签或模型输出类别 |
| `note` | 对候选来源、限制或原因的说明 |
| `status` | 人工审核状态 |

注意：不同模型的 `score` 不一定可直接比较。跨模型融合前需要校准。

## 4. Adapter 协议

每个 adapter 应实现：

```python
class SomeAdapter:
    name = "some_source"
    display_name = "Some Source"
    enabled = True
    out_of_domain = False

    def predict(self, case_id, volume_hu, metadata=None):
        ...
        return list[ExternalCandidate]
```

其中：

- `name`：程序内部使用的来源名；
- `display_name`：页面或日志中展示的名称；
- `enabled`：是否默认启用；
- `out_of_domain`：是否属于跨部位参考模型；
- `predict()`：接收 CT 体数据，返回统一候选点列表。

## 9. 当前版本的边界

当前只是预留接口和保存架构版本：

- 没有下载或集成任何外部模型权重；
- 没有运行肋骨/脊柱模型；
- 页面输出仍主要来自启发式规则；
- 外部模型占位 adapter 默认关闭；
- 还没有做跨模型分数校准。

这一步的价值是：后续可以按同一 adapter 规范逐步接入外部参考模型，而不破坏现有 demo 主流程。

## 11. nnInteractive 方向尝试记录

TotalSegmentator 暂时受 license 和 Windows native crash 阻塞后，下一条更适合 demo 阶段的路线是 nnInteractive。

nnInteractive 的定位不是自动骨折检测，而是 3D 医学图像交互式分割。它可以接受点、框、涂抹等提示，在体数据上生成 3D mask。对当前 demo 来说，它更适合作为：

- 医生点击疑似区域后的交互式 mask 生成器；
- 骨结构或局部疑似区域的分割参考；
- 后续人工标注与训练数据积累的辅助工具；
- 外部参考 adapter，而不是诊断模型。

### 12.3 当前问题和边界

当前仍需注意以下问题：

- nnInteractive 需要用户点击提示点，不能零输入自动检测骨折；
- 首次运行如果本地没有权重，需要从 Hugging Face 下载，耗时较长；
- 当前只展示 axial overlay，还没有把 mask 叠加到 coronal/sagittal；
- 当前还没有把 mask 转成 3D mesh 高亮；
- mask 质量依赖点击点，如果点到软组织或边界外，结果可能没有参考价值；
- 官方权重 license 为 `CC BY-NC-SA 4.0`，正式部署、商业交付或医院生产使用前必须重新核对授权；
- 当前 mask 输出没有写入候选点表格，也不会自动生成“骨折阳性”结论。

### 12.4 为什么不并入 Weak Candidates

`Weak Candidates` 表格表示程序主动给出的疑似候选点，而 nnInteractive 的输出是用户点击后的分割 mask。两者语义不同。

因此当前不把 nnInteractive 结果直接写入候选点表格，避免造成以下误解：

- 用户点击哪里，模型就“检测到骨折”；
- mask 覆盖区域就是骨折范围；
- nnInteractive 是骨折检测模型。

更合适的定位是：

```text
interactive_segmentation_reference
```

即“交互式分割参考”。

## 13. RSNA C-Spine 外部骨折模型接入方案

如果希望“找一个现成骨折检测模型，下载权重，接到 Gradio 看看效果”，当前建议优先尝试 RSNA 2022 Cervical Spine Fracture Detection 方向的公开推理方案，而不是先接肋骨或其它部位模型。

推荐候选：

```text
zxjasonhu/cspine-2022-challenge-inference-collection
```

参考入口：

```text
https://github.com/zxjasonhu/cspine-2022-challenge-inference-collection
```

该仓库整理了 RSNA 2022 颈椎骨折检测比赛的多个公开推理方案，并提供权重下载入口。它的医学域是颈椎 CT，不是手腕/足部 CT，因此只能作为跨域外部参考模型接入。

### 13.1 为什么优先选它

优先原因：

- 它是明确的 CT 骨折检测任务；
- 输入形态接近当前项目，都是 DICOM study；
- 仓库提供多个方案和权重入口；
- 输出通常是 study 级或椎体级 fracture probability，适合先展示在页面里；
- 部分方案带 Grad-CAM，可用于展示模型关注区域；
- 相比需要重新训练或缺少权重的肋骨模型，它更接近“下载权重后做推理验证”的目标。

当前不优先选择 FracNet / RibFrac Mask R-CNN 的原因：

- FracNet 方向偏肋骨骨折 3D segmentation，和手腕/足部域差异很大；
- 公开仓库需要额外确认权重、预处理和推理细节，直接跑通成本更高；
- RibFrac Mask R-CNN 相关仓库更偏训练/预测代码，不一定提供可直接用于推理的训练好骨折权重；
- 对当前 demo 来说，先证明“外部骨折模型可以被接入并展示输出”比追求模型域匹配更重要。

### 13.2 推荐先接哪个方案

建议先从该集合中的 7th-place / QWER 方案入手。

理由：

- README 中有成功复现记录；
- 通常比第一名大 ensemble 更容易落地；
- 不以最高分为目标，而以“能下载权重、能跑一次、能展示输出”为目标；
- 如果依赖过重或路径复杂，再降级选择仓库中更轻量的方案。

第一轮目标不是医学有效性，而是工程接入验证：

```text
DICOM study -> 外部模型推理 -> 输出预测分数/热力图 -> Gradio 展示
```

### 13.6 当前已知风险

主要风险：

- 颈椎模型对手腕/足部 CT 是严重 out-of-domain；
- 输出分数可能完全没有医学意义；
- 模型可能依赖 RSNA 数据集的目录结构和 metadata；
- 权重文件可能较大；
- 不同方案的依赖版本可能与 `fracmed` 当前包冲突；
- Grad-CAM 可能只适配原始方案的数据读取流程；
- 如果方案依赖旧版本 PyTorch / MONAI / timm，可能需要单独环境，而不是直接装进 `fracmed`。

因此建议采用隔离策略：

```text
先探针脚本验证 -> 再考虑是否新建 cspine 专用环境 -> 最后才接 Gradio
```

不要一开始把依赖直接装进 `fracmed`。

### 13.10 第一阶段发现的问题

7th/QWER 方案比普通分类模型更重。它不是简单的 2D 分类器，而是多阶段 3D nnU-Net 流程：

- Stage 1：C1-C7 椎体分割；
- Stage 2：骨折区域分割；
- Stage 3：根据分割结果生成最终分数。

该方案 README 声明：

```text
torch==1.11.0+cu115
Python 3.7+
At least 24 GB GPU memory
```

当前 `fracmed` 环境为：

```text
Python 3.10.19
torch 2.7.1+cu118
RTX 4060 Laptop GPU
```

因此不建议把 7th 方案的 `requirements.txt` 直接安装进 `fracmed`。原因：

- 它要求旧版 `torch` / `numpy` / `pandas` / `SimpleITK` / `pydicom` 等；
- 会和当前 nnInteractive、TotalSegmentator、Gradio 环境发生明显版本漂移；
- 7th 方案声明至少 24GB 显存，当前笔记本 GPU 很可能不足以完整推理；
- 即使权重下载完成，也可能需要单独环境或 CPU/低显存改造。
