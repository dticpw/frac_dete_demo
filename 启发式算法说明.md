# 启发式骨折候选算法说明

本文档说明当前 `frac_dete_demo` 中使用的启发式候选检测算法。

当前版本没有使用训练好的骨折检测模型，也没有使用医院标注数据。页面中的 `Weak Candidates` 来自一组基于 CT HU 值、骨结构阈值、连通域和边缘梯度的简单规则。

这些规则的目标是：

- 在没有训练数据时，先生成少量“值得人工查看”的候选点；
- 支撑 demo 阶段的交互流程；
- 帮助医生确认/否定候选，沉淀后续训练数据。

它们不是临床诊断算法，也不能代表最终模型能力。

## 1. 相关代码文件

当前算法主要分布在以下文件：

```text
src/config.py
src/volume_processing.py
src/candidate_detection.py
src/mesh_preview.py
```

其中：

- `config.py`：保存全局参数；
- `volume_processing.py`：负责骨窗和骨结构 mask；
- `candidate_detection.py`：负责生成疑似候选点；
- `mesh_preview.py`：负责 3D 骨结构预览，不直接决定候选点。

## 2. 核心参数

当前关键参数如下：

```python
DEFAULT_WINDOW_CENTER = 500
DEFAULT_WINDOW_WIDTH = 2000
BONE_THRESHOLD_HU = 350
MAX_CANDIDATES = 12
```

含义：

| 参数 | 当前值 | 含义 |
| --- | ---: | --- |
| `DEFAULT_WINDOW_CENTER` | `500` | 默认骨窗窗位 |
| `DEFAULT_WINDOW_WIDTH` | `2000` | 默认骨窗窗宽 |
| `BONE_THRESHOLD_HU` | `350` | 认为是骨结构的 HU 阈值 |
| `MAX_CANDIDATES` | `12` | 每个病例最多展示 12 个候选点 |

## 3. 预处理：骨结构 mask

候选检测的第一步是从 CT 体数据中提取骨结构区域。

代码位置：

```python
src/volume_processing.py
```

当前实现：

```python
mask = volume_hu >= BONE_THRESHOLD_HU
mask = ndi.binary_opening(mask, structure=np.ones((1, 2, 2)), iterations=1)
```

步骤解释：

1. 对完整 CT 体数据使用 HU 阈值；
2. HU 大于等于 `350` 的体素被认为属于骨结构或高密度结构；
3. 对 mask 做一次二值开运算；
4. 开运算结构元素是 `(1, 2, 2)`，也就是主要在单张切片内做轻微清理，不跨很多层处理。

这个 mask 是后续两个候选规则的基础。

## 4. 候选规则 1：小高密度连通域

代码位置：

```python
_small_fragment_candidates()
```

候选编号格式：

```text
{case_id}_frag_{idx:04d}
```

例如：

```text
1_frag_0007
```

表示 Case 1 中第 7 个由小高密度连通域规则生成的候选。

### 4.1 算法步骤

1. 对骨结构 mask 做 3D 连通域标记：

```python
labels, count = ndi.label(mask)
```

2. 遍历每个连通域；
3. 计算连通域体素数量 `size`；
4. 只保留大小在以下范围内的连通域：

```python
8 <= size <= 800
```

5. 对保留下来的小连通域，取其包围盒中心作为候选点坐标：

```python
zc = (z_start + z_stop - 1) / 2
yc = (y_start + y_stop - 1) / 2
xc = (x_start + x_stop - 1) / 2
```

6. 记录该局部区域内的最高 HU 值 `local_max`；
7. 生成候选点。

### 4.2 为什么要找小连通域

部分骨折可能表现为：

- 小骨片；
- 撕脱样骨折碎片；
- 与主体骨结构稍微分离的高密度小块；
- 局部异常骨性结构。

这些结构在骨阈值 mask 中可能形成较小的连通域。

### 4.3 分数计算

当前分数计算公式：

```python
score = min(0.95, 0.35 + size / 1400 + max(local_max - 700, 0) / 3000)
```

解释：

- 基础分是 `0.35`；
- 连通域越大，分数略高；
- 局部最高 HU 越高，分数略高；
- 分数最大限制为 `0.95`。

注意：这个分数不是骨折概率，只是启发式排序分。

### 4.4 reason 字段

该规则生成的候选点会显示：

```text
small high-density component
```

含义是：

> 当前点来自一个较小的高密度连通区域。

## 5. 候选规则 2：骨窗边缘突变

代码位置：

```python
_edge_candidates()
```

候选编号格式：

```text
{case_id}_edge_{z:04d}
```

例如：

```text
1_edge_0492
```

表示 Case 1 中，在第 492 层附近，由边缘突变规则生成的候选。

### 5.1 算法步骤

该规则按轴位切片处理。

1. 为了减少计算量，不是每一层都检测，而是按固定步长抽样：

```python
sample_step = max(1, volume_hu.shape[0] // 80)
```

如果一个病例有 520 层，则步长约为：

```text
520 // 80 = 6
```

也就是大约每 6 层取一层进行边缘检测。

2. 对当前切片裁剪 HU 范围：

```python
np.clip(image, -1000, 2000)
```

3. 计算图像梯度：

```python
gy, gx = np.gradient(...)
grad = np.hypot(gx, gy)
```

4. 只保留骨结构 mask 内部的梯度：

```python
edge = grad * local_mask
```

5. 取当前切片中骨结构梯度的极高分位数：

```python
threshold = np.percentile(edge[edge > 0], 99.7)
```

6. 找到超过该阈值的点；
7. 从这些点中取中间一个点作为候选；
8. 根据梯度强度生成分数。

### 5.2 为什么要找边缘突变

骨折可能表现为：

- 骨皮质中断；
- 骨边缘突然改变；
- 局部锐利断裂边界；
- 小骨片边缘。

这些位置在骨窗图像中通常有较强的灰度梯度。

但正常骨皮质边缘也会产生强梯度，所以这个规则误报会比较多。

### 5.3 分数计算

当前分数计算公式：

```python
score = min(0.9, edge_value / 2200)
```

并且过滤掉较低分数：

```python
if score < 0.35:
    continue
```

解释：

- 梯度越强，分数越高；
- 分数最大限制为 `0.9`；
- 小于 `0.35` 的候选直接丢弃。

注意：这个分数同样不是骨折概率。

### 5.4 reason 字段

该规则生成的候选点会显示：

```text
sharp bone-window edge change
```

含义是：

> 当前点来自骨窗图像中的强边缘变化。

## 6. 候选合并与去重

两个规则生成的候选点会合并到一起：

```python
small_fragments + edge_candidates
```

然后执行去重：

```python
_deduplicate(...)
```

当前去重规则：

```python
abs(cand.z - existing.z) <= 6
abs(cand.x - existing.x) <= 18
abs(cand.y - existing.y) <= 18
```

也就是说，如果两个候选在空间上很接近，就只保留分数更高的一个。

最后按分数从高到低排序：

```python
merged.sort(key=lambda item: item.score, reverse=True)
```

并最多保留：

```python
MAX_CANDIDATES = 12
```

## 7. 候选点输出格式

每个候选点使用 `Candidate` 数据结构：

```python
Candidate(
    candidate_id="1_edge_0492",
    case_id="1",
    slice_index=492,
    x=147,
    y=52,
    z=492,
    score=0.82,
    reason="sharp bone-window edge change",
    status="unreviewed",
)
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `candidate_id` | 候选点编号 |
| `case_id` | 病例编号 |
| `slice_index` | 候选点所在轴位切片 |
| `x` | 图像左右方向坐标 |
| `y` | 图像上下方向坐标 |
| `z` | 三维体数据中的 z 坐标 |
| `score` | 启发式排序分 |
| `reason` | 候选来源原因 |
| `status` | 人工审核状态 |

## 8. 3D 预览算法

3D 预览不直接影响候选点生成，它只是把骨结构和候选点可视化出来。

代码位置：

```python
src/mesh_preview.py
```

当前参数：

```python
MESH_VERSION = "v2"
MESH_MAX_DIM = 260
MESH_MAX_FACES = 400000
```

### 8.1 生成步骤

1. 根据体数据尺寸计算降采样倍率：

```python
factor = ceil(max(shape) / 260)
```

2. 对 CT 体数据按该倍率降采样：

```python
sampled = volume_hu[::factor, ::factor, ::factor]
```

3. 使用同样的骨阈值：

```python
mask = sampled >= 350
```

4. 做轻微开运算和闭运算；
5. 使用 marching cubes 提取骨结构表面网格：

```python
measure.marching_cubes(mask.astype(np.uint8), level=0.5)
```

6. 如果面数超过 `400000`，按步长抽样限制面数；
7. 写出 OBJ 文件；
8. 把候选点按降采样倍率映射到 3D 坐标中，并用小八面体作为标记。

### 8.2 缓存策略

生成后的 OBJ 会缓存到：

```text
data/cache/mesh/
```

文件名类似：

```text
case_2_bone_preview_v2.obj
```

后续再次加载同一病例时，会直接读取缓存文件。

## 9. 当前算法的主要局限

当前启发式算法很粗糙，主要局限包括：

- 没有学习真实骨折特征；
- 没有使用标注数据；
- 不理解具体解剖部位；
- 不区分正常骨皮质边缘和骨折边缘；
- 对正常小骨、籽骨、骨骺、骨缝可能误报；
- 对细微裂纹、非移位骨折可能漏检；
- 边缘规则目前按切片抽样，不是每层都完整检测；
- 小连通域规则对阈值和形态学处理比较敏感；
- 分数不是概率，没有经过校准；
- 不适合作为临床判断依据。

## 10. 为什么当前结果不准

当前结果不准是预期现象，原因是：

1. 规则只看 HU、连通域和梯度，不懂医学语义；
2. 骨头本身天然有很多强边缘；
3. 足、踝、腕、肘的小骨结构复杂；
4. 正常解剖变异容易被规则当作异常；
5. 没有医生标注数据来训练或校准；
6. 没有基于三维连续性的复杂判断。

因此，当前算法适合 demo 阶段展示“候选提示和标注闭环”，不适合证明检测准确率。

## 11. 后续可改进方向

短期改进：

- 增加候选缩略图；
- 对每个候选显示上下相邻切片；
- 把边缘规则改成局部 patch 分析；
- 增加基于骨表面曲率的候选；
- 增加按部位调整阈值；
- 添加人工筛选后的误报分析。

中期改进：

- 用医生确认/否定结果训练 patch 分类器；
- 将候选点周围 2.5D 或 3D patch 作为输入；
- 让模型学习区分正常骨边缘和疑似骨折。

长期改进：

- 使用 3D detection 或 segmentation 模型；
- 使用 MONAI / nnU-Net 等医学影像框架；
- 建立医院内部验证集；
- 用灵敏度、误报数、定位误差等指标评估。

## 12. 建议对外表述

对医院或合作方介绍时，建议这样说：

> 当前 demo 使用的是无训练数据条件下的启发式候选提示算法，主要用于验证 DICOM 读取、三视图浏览、3D 预览、候选提示和医生标注闭环。候选结果不是最终诊断结论。后续需要基于医生标注数据训练模型，才能逐步提升检测性能。
